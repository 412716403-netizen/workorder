# Project Analysis

## 目标

这个 Skill 用来分析当前仓库，并且通过明确的流程和质量闸门，尽量保证分析结果可信、可复查、可继续拆分。

默认应该帮助用户回答：

- 项目现在处于什么状态？
- 有没有严重问题或关键风险？
- 当前项目或某个模块是否需要修改、重构、补测试或继续迁移？

## 仓库上下文

把这个仓库视为一个制造业 / ERP / 生产进度报工系统，既有业务复杂度，也有技术迁移复杂度。

已知上下文：

- 前端：React + Vite + TypeScript
- 后端：Express + TypeScript + Prisma + PostgreSQL
- 业务与规则文档主要在 `docs/`
- 后端启动、API、环境说明主要在 `backend/README.md`
- 当前仓库看起来正处于从前端本地状态 / `localStorage` 逻辑逐步迁移到后端 API 和数据库持久化的阶段

分析时不要只看“现在代码怎么跑”，还要同时看：

- 当前状态
- 目标状态
- 两者之间的迁移缺口

## 阶段 1：规模评估与分析策略选择

在深挖之前，先估算项目规模并选择分析策略。

至少检查：

- 仓库目录结构和主要顶层模块
- 包清单与主要技术栈
- 是否存在后端、数据库层、迁移记录、业务文档、部署或配置资产
- 用户当前需求更偏向全局分析、模块分析、风险分析、架构分析，还是改造决策支持

把任务归类为以下模式之一：

1. 快速项目体检
2. 指定模块深度分析
3. 架构与数据流分析
4. 迁移 / 重构准备度评估
5. 严重问题 / 风险调查

明确说出选择了哪种模式，以及为什么这样选。

建议策略：

- 请求很聚焦：直接分析目标模块及其直接依赖
- 请求跨层级：按技术层或业务域拆开
- 请求覆盖整个仓库：先看文档和入口，再拆并行轨道
- 请求与迁移强相关：明确对比前端状态归属、后端接口归属、数据库 schema 归属

## 阶段 2：特征识别与自适应提问

第一轮扫完以后，不要立刻开始输出结论。先根据这个项目的特征继续追问，确认用户到底想看什么。

问题必须和当前仓库特性相关，避免泛泛而谈。

这个仓库常见的追问维度包括：

- 业务重点在哪：计划、工单、报工、库存、财务、打印、账号管理
- 用户更想看产品行为、代码质量、架构关系、迁移状态，还是改造建议
- 关注点是正确性、可维护性、性能、数据一致性、权限、交付风险还是用户体验
- 只分析前端、只分析后端，还是前后端一起看
- 是先要“理解现状”，还是直接要“改造建议”

结合本项目可优先提这些问题：

- 你更关心生产流程正确性、数据一致性，还是代码可维护性？
- 你要我分析当前前端本地逻辑、正在推进的后端迁移路径，还是两者之间的不一致？
- 打印模板、标签生成、预览链路是否在本次范围内？
- 你更想看业务规则是否正确，还是工程风险，比如耦合、重复、边界混乱？
- 你要先做全局健康检查，还是直接深挖某个模块？

如果用户目标仍然模糊，就继续多轮对话，直到范围足够清晰。不要假装已经理解了模糊需求。

## 阶段 3：生成分析规划

目标明确后，生成可以跨上下文窗口生存的分析计划。

计划必须拆成独立轨道，方便后续并行分析。

这个仓库推荐的拆分维度：

- 业务文档与规则来源：`docs/`
- 前端应用骨架、视图和用户流程
- 共享 `services`、`types`、`utils`
- 打印 / 标签 / 预览 / 导出链路
- 后端 routes、controllers、服务边界
- Prisma schema、migrations、数据持久化模型
- 前端状态到后端 / 数据库的迁移缺口

每条轨道都要定义：

- 目标
- 范围目录或文件
- 关键问题
- 预期产出
- 校验点
- 预估覆盖目标

轨道必须足够小，让子 agent 可以在隔离上下文中专注完成。

### 规划模板

默认使用这个结构：

```markdown
## Analysis Plan

### Track 1: [名称]
- Objective:
- Scope:
- Key questions:
- Expected output:
- Coverage target:

### Track 2: [名称]
- Objective:
- Scope:
- Key questions:
- Expected output:
- Coverage target:
```

## 阶段 4：Agent Teams 并行深度分析

当工作天然可以拆分时，优先并行。

使用 Agent Teams 还是 Subagent，由主 agent 根据规模自适应决定：

- 2 到 4 条清晰轨道：优先用有明确职责的 subagent
- 覆盖整个仓库的大分析：使用更宽的 agent team 式分工
- 每个子 agent 只负责一个模块、一个层级，或者一组非常清晰的问题

给并行 worker 的任务里必须包含：

- 精确范围
- 明确问题
- 预期输出格式
- 显式覆盖率要求
- 必须列出“看了什么”和“没看什么”

每个 worker 至少返回：

- 实际查看的文件或目录
- 发现的主要职责、链路或数据流
- 重要风险或严重问题
- 未确认的假设
- 信心等级
- 对其负责范围的估算覆盖率

## 阶段 5：交叉验证与质量管控

这是强制环节。不要接受“重点代码都分析完了”这种模糊说法。

主 agent 必须量化覆盖率，并在证据不足时强制返工。

每条轨道都要统计并报告：

- 计划范围
- 已检查范围
- 是否覆盖关键入口
- 大致文件 / 代码覆盖比例
- 是否覆盖了相关的文档、schema、routes、状态边界、核心 views
- 结论是否有直接代码证据支撑

默认质量门槛：

- 快速体检：必须看完主要顶层入口、包清单、核心文档
- 模块深挖：目标区域应覆盖大多数关键文件；没有合理排除理由时，尽量达到 80% 以上有效范围覆盖
- 架构 / 迁移分析：必须覆盖所有关键架构文件，并且每个主要模块至少抽样一个代表性实现
- 声称“存在严重问题”：最好有多源证据互相印证，例如文档 + 代码，或 schema + route + UI 路径

如果某个 worker 返回的结果证据弱、范围不清、覆盖率过低：

1. 明确指出缺口是什么
2. 用更窄、更具体的任务把它打回去重做
3. 在补齐前不要把该结论写进最终报告

覆盖率报告宁可粗略但明确，也不要假装精确。

好例子：

- “我检查了这个功能的 route 注册、对应 controller 和 Prisma schema，后端链路大约覆盖了 85%。”

坏例子：

- “我看过这块后端重点代码了。”

## 阶段 6：多源融合与最终报告

最终报告要融合：

- 仓库文档
- 直接代码阅读结果
- 并行 worker 的返回
- 用户最初提出的分析目标

最终报告不能只是笔记堆砌，而要能支持用户做判断。

### 最终报告模板

默认使用这个结构：

```markdown
# Project Analysis Report

## Scope and goal
- What was analyzed
- What was intentionally excluded

## Current state
- Architecture snapshot
- Business/domain snapshot
- Migration stage if relevant

## Serious issues and risks
- High-severity findings
- Why they matter
- Evidence basis

## Does it need modification?
- Yes / No / Partially
- What kind of modification is needed
- Priority order

## Recommended next actions
- Immediate
- Near-term
- Optional

## Coverage and confidence
- Areas inspected
- Approximate coverage by track
- Remaining blind spots
```

## 运行规则

- 不要夸大覆盖率
- 不要把“看起来重要的几个文件”误当成“已经有代表性的覆盖”
- 需求模糊时，不要跳过自适应提问
- 能拆并行时，不要把所有工作塞进单线程分析
- 不要把它做成纯代码风格 review；要同时看业务流、数据归属、迁移边界
- 如果用户只关心一个模块，就把流程缩轻，不要强行铺满整个仓库

## 当前仓库的高优先风险点

当不确定先看哪里时，优先检查：

- 业务规则文档与真实实现是否漂移
- 前端 `localStorage` / 持久化状态与后端数据真源迁移是否冲突
- Prisma schema 与 migrations 是否一致、是否安全
- 新增后端功能的 route / controller / 数据层链路是否闭环
- 打印、标签、预览、导出这类业务输出链路是否存在静默失败风险
- `views`、`services`、共享 `types`、后端接口契约之间是否一致

## 成功标准

这个 Skill 成功的标志是：

- 分析真正对准用户关心的问题
- 范围、盲区、覆盖率都说清楚
- 项目大时可以自然拆分并行
- 能抑制子 agent 偷懒和糊弄
- 最终结论足够支撑“要不要改、先改哪里”的决策
